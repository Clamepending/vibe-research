## 2026-04-19 — negation-arithmetic-sonnet-4-6

Tested the hypothesis that `claude-sonnet-4-6` (via `claude -p --model claude-sonnet-4-6 --tools ""`) systematically fails on arithmetic word problems that involve negation ("How many did NOT do X?"). Ran 3 probe cycles totaling 64 distinct probes × 3 runs = 216 evaluations spanning classic subtraction-negation, reverse (given-is-negated), double/triple negation, multi-subset exclusion, pragmatic ("hardly any", "few", "rarely"), scope ("not every", "not all"), neither/nor, "except for", buried-in-narrative, terse, bureaucratic ("failure to", "absence of"), and adversarial lures (trap-number reinforced, long-context with distractors, "none failed to submit", "did not not agree"). **Result: 214/216 correct (99.1%), zero surface-trap answers.** The two failures were both arithmetic-bookkeeping slips on the same multi-bucket probe (sedans/SUVs/trucks/convertibles), not negation misreads. Hypothesis decisively not supported. Full write-up in [experiments/negation-arithmetic-sonnet-4-6/README.md](experiments/negation-arithmetic-sonnet-4-6/README.md). Project repo at `/Users/mark/Desktop/projects/negation-arithmetic-probe`, tag `exp/negation-sonnet-4-6/conclusion`.

Key cross-cutting observation: when the "answer with just a number" suffix is removed, Sonnet 4.6 reliably *shows the arithmetic* (`"30 − 18 = **12 students** did not bring their lunch."`), which is strong evidence of genuine comprehension rather than lucky pattern-matching. Also verified with `--system-prompt ""` that the Claude Code default system prompt is not carrying the correctness.

## 2026-04-19 — code-review-prompt-tuning

Tuned a system prompt for `claude-sonnet-4-6` (via `claude -p --model claude-sonnet-4-6 --tools ""`) to produce substantive code-review comments on a buggy Python payment validator function. Ran baseline + 4 variants × 4 samples each across 3 experiment cycles (17 total generations). Scored each sample on a 6-issue ground-truth rubric committed to before any run. **Winner: V2 `variants/v2_checklist.txt` (tag `exp/code-review/best`, commit `1d52fa3`)** — a named-category checklist prompt (semantics / numeric correctness / input validation / API design / hidden state / domain correctness / concurrency). V2 hit **6/6 on every sample (4/4)**; baseline averaged 4.75/6, senior-persona V1 5.0/6, adversarial V3 4.25/6, minimal V4 4.25/6. Boilerplate rate 0% across all 20 samples — the "add type hints / consider error handling" baseline failure mode the task premise assumed did not occur on this model/input. Full write-up in [experiments/code-review-prompt-tuning/README.md](experiments/code-review-prompt-tuning/README.md). Project repo at `/Users/mark/Desktop/projects/code-review-prompt-tuning`, branch `experiment/wave1`, main branch empty.

Key cross-cutting findings: (a) On already-strong models, **category priming is the lever** — naming dimensions to check retrieves different slices of training distribution in a way that anti-boilerplate directives alone (V4) do not. (b) **Persona cues shift emphasis, not coverage** — V1 "senior engineer reviewing a PR" adds TOCTOU/concurrency observations in 3/4 samples (baseline: 0/4) but doesn't raise core-rubric coverage meaningfully. (c) **Asking for subtlety can reduce coverage** — V3's "find what a junior would miss" framing is interpreted as specificity-narrowing, making the model skip the loud semantic bugs in 3/4 samples. (d) For this model on this input, the baseline is already substantive — the prompt-craft lever size is perhaps 1.25 points on a 6-point rubric (baseline 4.75 → winner 6.0), not a 0 → 5 jump. Pattern worth testing on a weaker model (Haiku 4) and on different bug-distribution functions.

## 2026-04-19 — horror-prompt-tuning

Tuned a system prompt for `claude-sonnet-4-6` (via `claude -p --model claude-sonnet-4-6 --tools ""`) on two-sentence horror stories, user message fixed to "Write a two-sentence horror story." Ran baseline + 5 variants × 6–8 samples each across 5 experiment cycles. **Winner: `v4_composite.txt` (tag `exp/horror/best`, commit `64369a6`)** — a prompt that both gives a positive anchor (ground in a specific mundane physical particular) AND bans multiple failure modes (formulaic openers, twist-reveal structure, stock imagery, soup-on-stove, abstract emotion words, meta-output). 13/14 samples strong; no mode collapse; no prompt leak. Full write-up in [experiments/horror-prompt-tuning/README.md](experiments/horror-prompt-tuning/README.md). Project repo at `/Users/mark/Desktop/projects/horror-prompt-tuning`, branch `experiment/wave1`.

Key cross-cutting finding: **creative-genre prompts have multiple competing stereotype attractors in the base model**, not one. Baseline Sonnet 4.6 on "write a two-sentence horror story" collapses into twist-reveal slop ("The last thing she heard before the lights went out was…" in 4/6 samples, "She lived alone" in 3/6). Restraint framing alone shifts collapse to *soup-on-stove* (5/6). Ban list alone shifts collapse to *thirty-year-career vocation* (5/6). Only a composite that names and bans several attractors *and* anchors to a positive concrete particular produces varied, high-quality outputs. This likely generalises to other creative genres (flash comedy, haiku, aphorism) and is a candidate for a topic page once a second genre confirms the pattern.

## 2026-04-19 — uncertainty-admission-prompt-tuning

Can prompt-craft make `claude-sonnet-4-6` (via `claude -p --model claude-sonnet-4-6 --tools ""`) reliably admit uncertainty instead of confabulating? Built a 20-probe set across 6 categories (fabricated essays/papers, obscure-real-answerable controls, obscure-real-hard targets, common misconceptions, post-cutoff events, fake technical APIs). Ran baseline + six variant system prompts × n=3, LLM-judged by `claude-opus-4-7` with the probe author's ground-truth notes, plus an n=3 rerun of the winner for variance. **Winner: `v6_focused` (tag `exp/uncertainty/best`, commit `78d2298`, project repo `/Users/mark/Desktop/projects/uncertainty-admission-prompt-tuning`)** — an epistemic-fact-checker role with an explicit numeric-fact clause. Scored 60/60 on two independent runs (120 total samples, 0 confabulations). Baseline 90% correct; best prior variant (v5 epistemic-role) 98.3%; v6 100% on this probe set. Full write-up in [experiments/uncertainty-admission-prompt-tuning/README.md](experiments/uncertainty-admission-prompt-tuning/README.md).

Key cross-cutting findings: (a) **Frontier-model baseline is already strong** — sonnet-4-6 rejected every fabricated paper/essay/API I constructed, and correctly addressed every common misconception, post-cutoff event, and fake technical flag. The "makes up plausible-sounding answers to fake questions" failure mode the task premise assumed is already largely solved. (b) **The real residual failure is epistemic-status confabulation on specific-numeric facts**: the model has the right gist ("francium melts near room temp", "Scoraig has few dozen residents") but asserts specific numbers with false confidence ("27°C" without noting it's an extrapolation, "2011 census: 68" without a real census entry at that granularity). (c) **A numeric-fact epistemic clause — "for numbers, say whether it's a measurement you recall or an extrapolation/gist" — closes that gap**, where generic permission ("it's OK to say IDK") only helps at the margin. (d) **Structured confidence tags can backfire** — v4's `**High confidence.**` prefix becomes decorative prose, not a real reflection step. (e) **A rule-based scorer is wrong for this task** — it flagged "correctly rejects false premise but adds related true context" as CONFABULATION; LLM-as-judge with the probe author's ground-truth notes was required. (f) Caveat: 20 probes is small; two of them (p07 Scoraig, p18 francium) drove most of the baseline-vs-v6 delta, and v6 was designed partly in response to those specific failures, so there's overfitting risk. Honest prompt-tuning upper bound on this already-strong model is ~6-10pp of failure reduction, not 0 → 50.
